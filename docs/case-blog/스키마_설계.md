# B안 — 3단계 하이브리드 JSON 스키마

각 단계 출력은 JSON(구조) + 마크다운(내용). 사람이 검수·수정한 뒤
다음 단계 입력으로 넘긴다.

전체 흐름:
1단계 호출 → causalFlow JSON → 검수·수정 → 2단계 입력
2단계 호출 → outline JSON → 검수·수정 → 3단계 입력
3단계 호출 → blogPost JSON → 검수·수정 → 최종 저장

---

## 1단계 출력 — causalFlow (인과 흐름)

```json
{
  "phases": [
    {
      "id": "phase_1",
      "name": "단계명 (예: 수술 전 안정화)",
      "period": "경과 시점 (예: 첫 내원 시점 / 약 한 달 후) — 러프하게",
      "type": "surgical | medical | diagnostic",
      "what": "무슨 일이 있었나 (차트 근거, 마크다운 가능)",
      "why": "왜 이 처치/판단으로 이어졌나 (임상 원리)",
      "toNext": "다음 단계로 넘어간 계기 (마지막 단계는 빈 문자열)"
    }
  ],
  "axis": "이 케이스 흐름의 축 한 줄 요약 (CASE_OVERVIEW 기준)",
  "anesthesia": true
}
```

- `type`: 각 단계 성격. medical이면 시점별 변화 추적, surgical이면 처치 인과.
- `anesthesia`: 마취 동반 여부. true면 2단계에서 마취 안전성 검사 비중↑
- 검수 UI: phases 배열을 카드 목록으로. 순서 변경·내용 수정·단계 추가/삭제 가능하게.

---

## 2단계 출력 — outline (섹션 아웃라인)

```json
{
  "title_candidates": ["제목 후보 1", "제목 후보 2"],
  "sections": [
    {
      "id": "sec_1",
      "label": "아웃라인 섹션명 (예: 내원 배경)",
      "points": ["요점 불릿 1", "요점 불릿 2"],
      "facts": ["팩트 불릿 1 (검사 수치·검사명 등)", "팩트 불릿 2"]
    }
  ],
  "overviewCheck": [
    { "item": "CASE_OVERVIEW 항목", "reflectedIn": "sec_3" }
  ]
}
```

- 치료 과정이 여러 국면이면 sections에 "치료 과정 ①", "치료 과정 ②"로 여러 개.
- `points` = 서술 방향, `facts` = 반드시 들어갈 구체 데이터 (검사=구체/처치=일반어).
- `overviewCheck` = 누락 점검표. 검수 시 유저가 빠진 항목 바로 확인.
- 검수 UI: 섹션별 카드, points/facts 각각 편집. 섹션 순서 변경·추가 가능.

---

## 3단계 출력 — blogPost (최종 글)

```json
{
  "title": "최종 제목",
  "excerpt": "요약 (네이버 블로그 미리보기용)",
  "bodyMarkdown": "## 내원 배경: 소제목\n\n본문...\n\n[사진: 설명]\n\n...",
  "tags": ["강아지 방광결석", "혈뇨", "..."],
  "charCount": 2480
}
```

- `bodyMarkdown`: 섹션명 라벨 헤더 + 줄글(불릿 없음) + [사진:] 표시.
- `charCount`: 모델이 센 공백 포함 글자 수 (2000~3000 목표 확인용).
- 기존 시스템의 blog_post 키(title/excerpt/bodyMarkdown/tags)와 호환 — charCount만 추가.
- 검수 UI: 본문 마크다운 편집창 + 제목/태그 필드.

---

## 단계 연결 시 주의

- 1→2: 검수·수정된 causalFlow JSON을 통째로 2단계 프롬프트의 입력으로 넣는다.
- 2→3: 검수·수정된 outline JSON을 통째로 3단계 프롬프트의 입력으로 넣는다.
- 각 단계는 "이전 단계 JSON"을 받아 처리하므로, 프롬프트에 입력 JSON 구조 설명 포함.
- 수정본이 다음 단계로 가야 하므로, 검수 후 "승인" 시점의 데이터를 넘길 것.
  (원본이 아니라 유저가 고친 버전)
